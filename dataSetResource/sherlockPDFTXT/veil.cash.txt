Issue L-1: Deployer shouldn't be assigned the owner-
ship role [ACKNOWLEDGED]
Source: https://github.com/sherlock-audit/2026-02-veil-cash-mar-2nd-2026/issues/11
This issue has been acknowledged by the team but won't be fixed at this time.

Summary
At deploy time, deployer is made the owner.

Vulnerability Detail
Assigning deployer an admin role isn't a recommended practice as deployer is an EOA.
Even if the role is later transferred to a multi-sig, this temporary role assignment can be
lead to issues if the private key is known to an attacker.

Impact
Existing leak of the private key can lead to a third party taking over the role maliciously.

Code Snippet
constructor(IVerifier _verifier2, IVerifier _verifier16, uint32 _levels, address
↪→ _hasher, IERC20 _usdc)

Ownable(msg.sender)

Tool Used
Manual Review

Recommendation
Take an explicit address _owner argument in constructor for the ownership role.

Discussion
apexethdev
Noted.

Vulnerability Detail

}

Manual Review

Add an ETH withdrawal event.

Issue L-4: Unused Constants [ACKNOWLEDGED]
Source: https://github.com/sherlock-audit/2026-02-veil-cash-mar-2nd-2026/issues/14
This issue has been acknowledged by the team but won't be fixed at this time.

Summary
VeilLib defines BASIS_POINTS and MerkleTreeWithHistory defines ZERO_VALUE, both are
never referenced.

Vulnerability Detail
VeilLib.BASIS_POINTS is declared but never used by any in-scope contract.
MerkleTreeWithHistory.ZERO_VALUE is declared but never referenced — the contract uses
the zeros(0) function to get the same value instead.

Impact
Dead code increases contract size and deployment gas cost.

Code Snippet
// src/lib/VeilLib.sol:55
uint256 public constant BASIS_POINTS = 10000;

// src/VeilPool/verifiers/MerkleTreeWithHistory.sol:14
uint256 public constant ZERO_VALUE =

Tool Used
Manual Review

Recommendation
Remove both unused constants.
