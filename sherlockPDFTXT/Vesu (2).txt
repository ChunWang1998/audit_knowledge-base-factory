Introduction
Vesu is a fully open and permissionless lending protocol built on Starknet. Users can
supply crypto assets (earn), borrow crypto assets and build new lending experiences on
Vesu without relying on intermediaries. The Vesu lending protocol is not controlled by a
governance body and there exists no governance token. Instead, Vesu is built as a public
infrastructure giving everyone equal access to all functions and is free for everyone to
use.

IssueM-1: Ekuboadapter feecollectionflagprevents
collection of pending fees on full withdrawal [RE-
SOLVED]
Source: https://github.com/sherlock-audit/2026-02-vesu-feb-4th/issues/134

Summary
The withdraw_liquidity(...) function in the Ekubo adapter does not collect pending
fees when withdrawing liquidity. When the withdrawal amount equals total liquidity, the
adapter clears the NFT identifier, making unclaimed fees irrecoverable.

Vulnerability Detail
The withdraw_liquidity(...) function calls the Ekubo contract's withdraw(...) function
with the fee collection flag hardcoded to false. This means any pending fees
accumulated in the position are not collected during withdrawal. If the withdrawal
amount equals total liquidity, the adapter clears the contract_nft_id storage variable.
Since unclaimed fees remain tied to the NFT and the contract_nft_id is discarded, these
fees become permanently unrecoverable.

Impact
Pending fees that have accumulated in the Ekubo position are permanently lost when
withdrawing the full liquidity amount. This results in direct loss of yield for the vault.

IssueM-3: Bridgemiddlewaresclaimingcanbegriefed
[RESOLVED]
Source: https://github.com/sherlock-audit/2026-02-vesu-feb-4th/issues/138

Summary
The bridges' middleware claim_token(...) function allows anyone to call it and clear a
request's pending_balance. This can be exploited by users when there are multiple
requests for a token_to_claim.

Vulnerability Detail
When there are multiple requests for a token_to_claim anyone can call the claim_token(.
..) function and use the current token_to_claim balance for any request he desires. For
example:
There are 2 bridging requests:

1. 1000 USDT for ETH
2. 100 USDC for ETH

The 1st request gets completed. Users can now call the claim_token(...) function for the
2nd request, which resets the mapping. When the 2nd request gets completed, the
amount of ETH will be too low to call claim_token(...) on the 1st request.
Additionally, if both requests get completed at the same time, it's only possible to call cl
aim_token(...) on only one of them.

Impact
Users can DOS certain requests and some requests can't be cleared, blocking the
creation of similar requests.

IssueL-1: Bridgemiddlewarescalculate theslippage
at the time of claiming, not at the time of the swap
[ACKNOWLEDGED]
Source: https://github.com/sherlock-audit/2026-02-vesu-feb-4th/issues/136
This issue has been acknowledged by the team but won't be fixed at this time.

Summary
The bridge middleware contracts implement two functions, a function that initiates the
bridging process and a claim_token(....) function. The claim_token(...) function uses
the current prices of the tokens to calculate the slippage and the minimum output
amount. If the contract's token balance is not greater than this amount, the transaction
reverts. However, the current prices of the tokens don't reflect the prices at the time of
the swap.

Vulnerability Detail
Price fluctuations between the swap and the completion of the bridging process can
make the claim_token(....) function revert. At these cases funds are temporarily stuck.
Admins must manually transfer tokens to the middleware contract to increase its token
balance, so the claim_token(....) function will not revert.

Impact
The claim_token(....) function can unnecessarily revert and admins must manually
transfer funds to rescue the locked tokens.

strategy configuration on each call, preventing independent permission assignments to
different strategies.

Vulnerability Detail
The set_vault_configs(...) function allows the governor to configure various strategies
including vesu_v2_configs, starknet_vault_kit_strategies, and others. However, the
permission control mechanism lacks flexibility because each call invokes vault_aum_prov
ider.set_vault_config(...), which overwrites the previous strategy configuration

+ starknet_vault_kit_strategies, but they cannot have completely independent
permission sets.

Impact
This limitation reduces the flexibility of strategy management. In scenarios where
different strategies require independent permission sets that don't follow a subset
relationship, the protocol cannot accommodate such configurations.

tokens, potentially leading to unnecessary contract deployments and increased gas
