# Training Spec — Prompt Optimization via SkillOpt

## Original Requirements (from README Optimize section)

- 用 AI tool 來 train prompts 內容
- 用 knowledgebase 的一半的 issues 當作 train set，一半的 issues 當作 validation set，需要額外一些內容來當 test set
- prompt 不斷的被 train set 修改，拿去 validation set 評分，並且根據評分結果決定是否要回頭優化 prompts
- 目標：訓練過後的 prompts 可以找到 8 成 validation set（分數 80%），5 成 test set。先確定內部是優秀的，不要急著去對外部

Paper: SkillOpt — Executive Strategy for Self-Evolving Agent Skills (arXiv:2605.23904)

## Background: What SkillOpt Does

SkillOpt is a text-space optimizer that trains a deployable natural-language
"skill" (a .md / .txt prompt file) for a frozen LLM without touching model
weights. The loop is:

  Seed Skill
      │
      ▼  Run agent on train batch → collect trajectories (correct / wrong)
      │
      ▼  Optimizer LLM proposes ONE bounded edit
      │
      ▼  Evaluate candidate skill on val set
      │
      ├─ val improved? YES → Accept → new best skill
      └─ NO → Reject (optimizer memorizes rejected version)

Mapping to this project:
  - prompts/*.txt               = the "skill"
  - knowledgeBase/**/issues.json = the dataset (358 issues total)
  - Audit agent task            = given vulnerable code, identify vulnerability tags
  - Evaluator (to implement)    = tag recall
  - Train/Val/Test split        = 50% train / 50% val per category + external test set

### 2c. Replace Agent (skillopt/agent.py)

  BASE_INSTRUCTIONS = """
  You are a smart contract security auditor.
  Below is your skill (audit checklist). Apply it strictly.

  {skill}    ← injected from prompts/accounting.txt (or other category)
  """

  USER_TEMPLATE = """
  {question}

  Respond with:
  - Vulnerability Type: (exact tag from taxonomy)
  - Category: (e.g. accounting/payout-errors/hardcoded-overpayment, ≥3 layers)
  - Explanation: (1-2 sentences)
  """

The skill (prompts/*.txt) is injected into BASE_INSTRUCTIONS at runtime.
Loading mechanism: pass --seed-skill-file <path> to skillopt-train; the CLI
reads the file and sets cfg.seed_skill. Keep seed_skill: "" in the YAML.

max_tokens is raised to 1024 in audit configs (agent must emit Vulnerability
Type + Category path + Explanation, not just a single-line answer).

### 2d. Replace Evaluator (skillopt/evaluator.py) (DONE)

Two additions are made on top of the existing EM/F1 evaluator:

1. tag_recall() — word-boundary match (NOT bare substring) to avoid false
   positives. e.g. tag "reentrancy" must NOT match "reentrancy-guard":

     def tag_recall(predicted_text, tags) -> float:
         _BOUNDARY = r"(?<![a-z0-9_-])", r"(?![a-z0-9_-])"
         pred = predicted_text.lower()
         for tag in tags:
             tag_lower = tag.lower()
             for form in {tag_lower, tag_lower.replace("-", " ")}:
                 pattern = _BOUNDARY[0] + re.escape(form) + _BOUNDARY[1]
                 if re.search(pattern, pred):
                     return 1.0
         return 0.0

   Decision: two surface forms (hyphenated + space) + custom lookaround boundary.
   - Two forms: model may output "hardcoded-overpayment" or "hardcoded overpayment".
   - Custom boundary (?<![a-z0-9_-]) instead of \\b: \\b fires at the y/-
     junction in "reentrancy-guard" (hyphen is non-word char); explicit exclusion
     of [-] blocks that false positive.

2. evaluate_recall(trajectories) — sets t.correct via tag recall (not EM) so
   the optimizer's failure analysis focuses on missed vulnerability tags.
   Returns EvalResult with recall field populated.

3. EvalResult gains a recall: float = 0.0 field.
   EvalResult.metric() now accepts "recall" in addition to "em" / "f1".

4. The existing evaluate() still runs EM/F1 (HotpotQA backward compat) but
   also computes recall as a side-channel metric in the same pass.

trainer.py dispatches to evaluate_recall() when cfg.metric == "recall",
otherwise uses the original evaluate(). The evaluate_skill() function in
trainer.py also respects cfg.metric for test-set scoring.

Target: val recall ≥ 0.80 (80% of validation issues correctly tagged).

Note on patterns field: each issue also has a "patterns" field. This is
intentionally excluded from both the agent input and the ground truth. Tags
are the canonical ground truth; patterns are supplementary and would risk
over-specificity if used as additional eval targets.

### 2f. Update Optimizer (skillopt/optimizer.py) (DONE)

This file is NOT listed in the original spec but MUST be changed along with
agent.py and evaluator.py. The defaults are HotpotQA-specific:

  SEED_SKILL     — was QA ("identify which paragraphs are relevant…")
                   → now audit ("Inspect Solidity code, identify vulnerability
                     patterns, output the most specific vulnerability tag…")

  _OPTIMIZER_SYSTEM — was "frozen question-answering agent … multi-hop questions"
                   → now "frozen smart contract security auditor … vulnerability
                     patterns, taxonomy, edge cases"

Both are updated in skillopt/optimizer.py. The optimizer_max_tokens is raised
to 2048 in all audit configs (audit checklists are longer than QA skills).

## Step 3 — Training Loop (End-to-End)

  prompts/accounting.txt          ← seed skill (existing prompt)
          │
          ▼
  [Audit Agent] — runs on train batch (54 Accounting issues)
          │       skill injected as system context
          ▼
  [Trajectories] — correct (tag found) + wrong (tag missed)
          │
          ▼
  [Optimizer LLM] — sees failures, proposes ONE bounded edit
          │           e.g. "Add checklist item for fee-on-transfer edge cases"
          ▼
  [Val Gate] — run agent on 54 val issues with candidate skill
          │
          ├─ recall improved? YES → accept → new best_skill
          └─ NO → reject, tell optimizer not to repeat this edit
          │
          ▼
  skillopt/outputs/audit/accounting/best_skill.md   ← deployable result

Stopping: 4 consecutive rejections (train.patience) OR val recall ≥ 0.80.

## Step 4 — Test Set Strategy

Goal: 50% recall on external test set (unseen data).

Use audit reports NOT yet in knowledgeBase:
  - New Sherlock reports after current scope (post 2026.03) from dataSetResource/
  - New Cyfrin reports not yet ingested
  - Any fresh Hacken / Peckshield reports

Convert to same item format and store at:
  skillopt/data/audit/{category}/test/items.json

Test set must never be seen during train or val optimization.

## Step 6 — Key Design Decisions

| Decision                    | Recommendation                                              |
|-----------------------------|-------------------------------------------------------------|
| Monorepo vs separate clone? | Monorepo — skillopt/ is already vendored in this repo       |
| Data source                 | knowledgeBase/**/issues.json (not a flat index.json)        |
| One skill per category?     | YES — matches prompts/*.txt structure, cleaner signal       |
| Agent input                 | code_snippet + contract_name + functions_affected           |

| Optimizer model             | Stronger than target agent (GPT-4o, claude-opus)            |

| Stopping criterion          | 4 consecutive rejects OR val recall ≥ 0.80                  |
| Deploy                      | Copy best_skill.md → prompts/*.txt                          |

## 背景：SkillOpt 做了什麼

SkillOpt 是文字空間優化器，為凍結的 LLM 訓練可部署的自然語言「技能」（prompt
文字檔），不修改模型權重。訓練迴圈：

  種子技能（seed skill）
      │
      ▼  讓 agent 帶著技能跑 train batch，收集軌跡（答對/答錯）
      │
      ▼  Optimizer LLM 提出一次有界編輯
      │
      ▼  在 val set 評估候選技能
      │
      ├─ val 分數提升？YES → 接受，成為新的最佳技能
      └─ NO → 拒絕，並記憶這次被拒絕的版本

本專案對應：
  - prompts/*.txt                    = 「技能」
  - knowledgeBase/**/issues.json     = 資料集（共 358 筆 issues）
  - Audit agent 任務                 = 給定有漏洞的程式碼，找出漏洞 tags
  - 評分器（待實作）                 = tag recall（召回率）
  - 資料切分                         = 每類別 50% train / 50% val + 外部 test set

### 2d. 改寫評分器（skillopt/evaluator.py）（已完成）

在現有 EM/F1 評分器之上新增兩個函式：

1. tag_recall() — 使用詞邊界（word boundary）比對，避免假陽性。
   例如 tag "reentrancy" 不應命中 "reentrancy-guard"：

     def tag_recall(predicted_text, tags) -> float:
         _BOUNDARY = r"(?<![a-z0-9_-])", r"(?![a-z0-9_-])"
         pred = predicted_text.lower()
         for tag in tags:
             tag_lower = tag.lower()
             for form in {tag_lower, tag_lower.replace("-", " ")}:
                 pattern = _BOUNDARY[0] + re.escape(form) + _BOUNDARY[1]
                 if re.search(pattern, pred):
                     return 1.0
         return 0.0

   決策說明：兩種表面形式（帶連字號 + 空格）加上自定義 lookaround 邊界。
   - 兩種形式：模型可能輸出 "hardcoded-overpayment" 或 "hardcoded overpayment"。
   - 自定義邊界 (?<![a-z0-9_-]) 而非 \\b：\\b 在 "reentrancy-guard" 中的
     y/- 交界點仍會觸發（因為 - 是非詞字元）；明確排除 [-] 可避免假陽性。

2. evaluate_recall(trajectories) — 以 tag recall 設定 t.correct（非 EM），
   使 optimizer 的失敗分析聚焦於「漏掉的漏洞 tag」。
   回傳帶有 recall 欄位的 EvalResult。

3. EvalResult 新增 recall: float = 0.0 欄位。
   EvalResult.metric() 現在也接受 "recall"（除了原有的 "em"/"f1"）。

4. 現有 evaluate() 保留 EM/F1（維持 HotpotQA 向後相容），同時也計算 recall
   作為附帶指標。

trainer.py 在 cfg.metric == "recall" 時呼叫 evaluate_recall()，否則使用
原本的 evaluate()。evaluate_skill() 同樣遵循 cfg.metric 進行 test set 評分。

目標：val recall ≥ 0.80（validation set 80% 召回率）。

patterns 欄位說明：每筆 issue 另有 patterns 欄位，刻意不納入 agent 輸入
或 ground truth。tags 是標準答案；patterns 是補充資訊，若用作評分目標
可能導致過度特異化（over-specificity）。

## Step 6 — 關鍵設計決策

| 決策                 | 建議                                                        |
|----------------------|-------------------------------------------------------------|
| Monorepo vs 另 clone | Monorepo — skillopt/ 已內嵌在本 repo                        |
| 資料來源             | knowledgeBase/**/issues.json（非扁平 index.json）           |
| 每類別一個 skill？   | 是 — 對應 prompts/*.txt，優化信號更純淨                    |
| Agent 輸入           | code_snippet + contract_name + functions_affected           |
| Ground truth         | tags 列表 — 輸出命中任一 tag 即算成功                       |
| Optimizer 模型       | 比 target agent 更強（GPT-4o、claude-opus）                 |
| Batch size           | 每批 10~15 個 issues                                        |
| 停止條件             | 連續 4 次拒絕，或 val recall ≥ 0.80                          |
| 部署                 | 複製 best_skill.md → prompts/*.txt                          |

每次訓練產出：
  skillopt/outputs/audit/{category}/{run_name}/

    ├── skills/skill_vXXXX_*.md
    ├── steps/step_XXXX.json
    └── test_result.json     ← test set 最終 recall

最終 best_skill.md 可直接複製到 prompts/accounting.txt 等，在 Grok / Cursor
