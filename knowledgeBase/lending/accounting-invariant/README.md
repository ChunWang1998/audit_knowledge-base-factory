# lending / accounting-invariant

- Count: `6`
- Definition: invariant mismatch between accounting model and real balances, or wrong financial formula basis.

## [Beraborrow][M-10] InternalizeDonations fails for asset token
- Severity: `Medium`
- Source: [Issue #43](https://github.com/sherlock-audit/2025-01-boyco-judging/issues/43)
- Impact: `locked-funds`, `functional-break`

### Detailed Content
- Summary: donation internalization derived donated amount as `balanceOf(this) - virtualBalanceWithFutureEmissions`.
- Root Cause: accounting model assumes vault holds principal token balance, but for main asset funds are staked externally so on-contract balance can be low/zero while virtual accounting is high.
- Trigger Conditions: any donation in this state makes subtraction underflow/revert in donation handling path.
- Impact Detail: donated assets become stuck and donation-related functionality breaks.

### Fix Status
- `Fixed/Resolved in report`

## [Beraborrow][M-15] Double performance fee on donations
- Severity: `Medium`
- Source: [Issue #125](https://github.com/sherlock-audit/2025-01-boyco-judging/issues/125)
- Impact: `fund-loss`, `value-mispricing`

### Detailed Content
- Summary: token donated without oracle is later internalized after oracle mapping, and fee logic executes twice across these phases.
- Root Cause: donation and internalization pathways both include performance fee charging without state guard preventing duplicate fee application.
- Attack/Trigger Path: donate token with no oracle -> later add oracle -> call internalization -> second fee charged.
- Impact Detail: escrow receives less than expected due to double haircut, creating value mis-accounting.

### Fix Status
- `Fixed/Resolved in report`

- Summary: withdrawal request valuation used yield-token pricing even when conversion result was fixed at cooldown initiation.
- Root Cause: valuation path did not distinguish slashable/variable LST-like requests from deterministic cooldown assets (e.g. USDe-style flows).
- Trigger Conditions: request created where shares are burned and fixed underlying amount is escrowed; later valuation still references token market price.
- Impact Detail: systematic mispricing of withdrawal requests affects collateral valuation and downstream risk/accounting decisions.

## [USG-Tangent][M-12] Liquidation fee computed from repaid debt
- Severity: `Medium`
- Source: [Issue #651](https://github.com/sherlock-audit/2025-08-usg-tangent-judging/issues/651)
- Impact: `value-mispricing`, `functional-break`

### Detailed Content
- Summary: liquidation fee computed as percentage of `USGToRepay` instead of liquidation bonus/profit.
- Root Cause: formula in `_liquidate` applies fee to debt repayment principal, not to surplus value generated by liquidation.
- Trigger Conditions: partial/full liquidation where fee portion becomes large relative to expected liquidator margin.
- Impact Detail: profitable liquidations can become loss-making, weakening liquidator incentives and protocol solvency operations.

## [USG-Tangent][M-14] Edge-case USG prices can force IR reverts
- Severity: `Medium`
- Source: [Issue #727](https://github.com/sherlock-audit/2025-08-usg-tangent-judging/issues/727)
- Impact: `dos`

### Detailed Content
- Summary: `_computeIR()` performs integer division before fixed-point conversion, truncating edge-case ratios to zero.
- Root Cause: loss of precision in quotient calculation feeds `_pow()` with zero and reaches `ABDKMath64x64.log_2(0)` revert path.
- Trigger Conditions: oracle returns USG price very close to upper bound (`pMax * 1e12`) such that numerator/denominator integer division yields zero.
- Impact Detail: IR-dependent actions (borrow/repay/withdraw/liquidate/migrate) can revert, creating protocol DoS windows.

- `Fixed/Resolved in report`
