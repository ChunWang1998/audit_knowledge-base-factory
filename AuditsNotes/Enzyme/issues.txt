Issue 1
knowledgeBase type: griefing-attacks/withdrawal-griefing/batch-processing/revert-in-loop
The main reference issue title in knowledgeBase: Permanent lock of withdrawal queue due to one malicious entry that always reverts.
File Location: src/components/ccip/WalletsManager.sol (function batchSendTokensViaCCIP)
Issue Description: The admin-only batch function for cross-chain token returns (the protocol’s withdrawal/return path) loops over wallets without any per-iteration try-catch, skip logic, or error handling. A single failing wallet causes the entire batch transaction to revert, blocking legitimate returns for all other wallets.
Exact code proving the vulnerability:
solidityfor (uint256 i; i < _params.length; i++) {
    DepositorWallet wallet = DepositorWallet(payable(_params[i].wallet));

    (Client.EVM2AnyMessage memory message, uint256 fee) = wallet.buildTokenReturnMessage({
        _tokens: _params[i].tokens, _extraArgs: _params[i].extraArgs, _feeToken: address(0)
    });

    wallet.sendCCIPMessage{value: fee}({_message: message, _feeValue: fee});
}
Attack scenario: Attacker (via flash-loan or prior malicious deposit) configures one wallet to always revert in buildTokenReturnMessage or sendCCIPMessage (e.g., malicious token that reverts on balanceOf/approve). When the admin calls batchSendTokensViaCCIP to process multiple users’ legitimate withdrawals/returns, the entire tx reverts → griefing/delay of all cross-chain withdrawals (in-scope High: Temporary freezing of funds + Medium: Griefing). No profit motive required; pure liveness attack repeatable at low cost.
