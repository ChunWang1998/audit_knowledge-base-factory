Issue 1
knowledgeBase type: dos-liveness/revert-lock/token-trapping/ccip-token-transfer-loop
The main reference issue title in knowledgeBase: Missing safe external call wrapper → one bad token locks entire queue.
File Location: src/components/ccip/WalletsManager.sol (function _ccipReceive)
Issue Description: The token transfer loop happens before the try-catch on processMessageData and has no per-token try-catch or safe-skip logic. A single failing transfer permanently bricks the entire CCIP message handler for that message.
Exact code proving the vulnerability:
solidity// Transfer received tokens to the user's wallet
Client.EVMTokenAmount[] memory tokenAmounts = _message.destTokenAmounts;
for (uint256 i; i < tokenAmounts.length; i++) {
    IERC20(tokenAmounts[i].token).safeTransfer({to: wallet, value: tokenAmounts[i].amount});
}
Attack scenario: Attacker (unlimited flash-loans) crafts and sends a CCIP message containing one malicious/blacklisted token that reverts on safeTransfer to the deterministic wallet address. The entire _ccipReceive reverts → incoming user funds become temporarily/permanently frozen at WalletsManager (in-scope High: Temporary freezing of funds / Permanent freezing of funds). No profit motive required; pure griefing + liveness DoS.

Issue 2
knowledgeBase type: dos-liveness/revert-lock/batch-execution/execute-calls-revert-grief
The main reference issue title in knowledgeBase: Missing safe external call wrapper in batch execution (revert-lock/return-script pattern).
File Location: src/ccip/DepositorWallet.sol (function executeCalls)
Issue Description: The call loop uses low-level Address.functionCallWithValue with no per-call try-catch, gas limit, or skip logic. Any single reverting call DoS-es the entire batch.
Exact code proving the vulnerability:
solidityfunction executeCalls(Call[] calldata _calls) external onlyWalletsManager {
    for (uint256 i; i < _calls.length; i++) {
        Address.functionCallWithValue({target: _calls[i].target, data: _calls[i].data, value: _calls[i].value});
        emit CallExecuted({target: _calls[i].target, data: _calls[i].data, value: _calls[i].value});
    }
}
Attack scenario: Attacker sends a CCIP message whose decoded Call[] contains mostly valid calls + one always-reverting call. Legitimate operations (and any subsequent token-return logic) are completely blocked → griefing DoS on user/protocol liveness (in-scope Medium: Griefing).

Issue 3
knowledgeBase type: external-dependencies/stale-data/external-contract-interaction/ccip-router-fee-calculation
The main reference issue title in knowledgeBase: Oracle price used in critical calculations without max/min bounds or sanity checks.
File Location: src/ccip/DepositorWallet.sol (function buildTokenReturnMessage)
Issue Description: The CCIP fee is fetched directly from the external IRouterClient.getFee call and used in token-amount deduction + message construction with no upper-bound, sanity check, or max-fee validation.
Exact code proving the vulnerability:
solidityfee_ = IRouterClient(CCIP_ROUTER).getFee({destinationChainSelector: getChainSelector(), message: message_});
Attack scenario: Attacker (or CCIP router outage/malfunction) causes getFee to return an excessively large fee value → fee deduction underflows or leaves insufficient token amount for the return message, permanently/temporarily freezing return funds in the DepositorWallet (in-scope High: Permanent freezing of funds / Temporary freezing of funds). No huge capital required; pure external-dependency griefing.

Issue 4
knowledgeBase type: upgrade-config/initializer/access-control/missing-onlyproxy-guard
The main reference issue title in knowledgeBase: Missing onlyProxy / onlyInitializing guard on initialize function.
File Location: src/ccip/DepositorWallet.sol (function init)
Issue Description: The initializer function uses only the initializer modifier but lacks the standard onlyProxy guard. This allows direct calls to the implementation contract (common recommendation for beacon/UUPS implementations).
Exact code proving the vulnerability:
solidityfunction init(address _walletsManager, bytes memory _user, uint64 _chainSelector) external initializer {
    __setWalletsManager(_walletsManager);
    __setUser(_user);
    __setChainSelector(_chainSelector);
}
Attack scenario: Attacker calls init directly on the known implementation address (or in a misconfigured deployment/upgrade scenario) with malicious parameters → corrupts wallet state (walletsManager/user/chainSelector). Subsequent proxy operations revert or behave incorrectly → permanent/temporary freezing of user funds across wallets (in-scope High: Permanent freezing of funds / Temporary freezing of funds). No profit motive required; pure griefing + liveness DoS on upgrade path.

Issue 1
knowledgeBase type: griefing-attacks/withdrawal-griefing/batch-processing/revert-in-loop
The main reference issue title in knowledgeBase: Permanent lock of withdrawal queue due to one malicious entry that always reverts.
File Location: src/components/ccip/WalletsManager.sol (function batchSendTokensViaCCIP)
Issue Description: The admin-only batch function for cross-chain token returns (the protocol’s withdrawal/return path) loops over wallets without any per-iteration try-catch, skip logic, or error handling. A single failing wallet causes the entire batch transaction to revert, blocking legitimate returns for all other wallets.
Exact code proving the vulnerability:
solidityfor (uint256 i; i < _params.length; i++) {
    DepositorWallet wallet = DepositorWallet(payable(_params[i].wallet));

    (Client.EVM2AnyMessage memory message, uint256 fee) = wallet.buildTokenReturnMessage({
        _tokens: _params[i].tokens, _extraArgs: _params[i].extraArgs, _feeToken: address(0)
    });

    wallet.sendCCIPMessage{value: fee}({_message: message, _feeValue: fee});
}
Attack scenario: Attacker (via flash-loan or prior malicious deposit) configures one wallet to always revert in buildTokenReturnMessage or sendCCIPMessage (e.g., malicious token that reverts on balanceOf/approve). When the admin calls batchSendTokensViaCCIP to process multiple users’ legitimate withdrawals/returns, the entire tx reverts → griefing/delay of all cross-chain withdrawals (in-scope High: Temporary freezing of funds + Medium: Griefing). No profit motive required; pure liveness attack repeatable at low cost.
Issue 2
knowledgeBase type: griefing-attacks/withdrawal-griefing/token-return/malicious-token-revert
The main reference issue title in knowledgeBase: Griefing redeem functions by triggering reverts that leave queue in inconsistent state.
File Location: src/ccip/DepositorWallet.sol (function buildTokenReturnMessage)
Issue Description: The token-return (withdrawal) message builder loops over an admin-supplied token list and directly calls balanceOf with no try-catch or validation. One malicious/reverting token bricks the entire return-message construction for that wallet.
Exact code proving the vulnerability:
solidityClient.EVMTokenAmount[] memory tokenAmounts = new Client.EVMTokenAmount[](_tokens.length);
for (uint256 i; i < _tokens.length; i++) {
    tokenAmounts[i] =
        Client.EVMTokenAmount({token: _tokens[i], amount: IERC20(_tokens[i]).balanceOf(address(this))});
}
Attack scenario: Attacker forces a wallet (via CCIP deposit with a specially crafted token) to include a malicious token in the return list. When return/withdrawal is triggered (either via processMessageData or batchSendTokensViaCCIP), buildTokenReturnMessage reverts → the specific wallet’s funds cannot be returned cross-chain, permanently/temporarily freezing them and griefing the overall withdrawal flow (in-scope High: Temporary freezing of funds + Medium: Griefing).
