1. Title
Fee-on-Transfer Token Under-Transfer in _addCollateral via safeTransferFrom
with Exact Amount Causing Collateral Accounting Mismatch

2. Description

## Brief/Intro
SecuritizeWallet._addCollateral() calls safeTransferFrom(investor, wallet,
amount) and then immediately forceApprove(creditManager, amount) using the
same amount value. For fee-on-transfer tokens, the actual amount received by
the wallet is less than `amount`, while the approval granted to the
CreditManager is still for the full `amount`. This discrepancy causes the
CreditManager to attempt to pull more tokens than the wallet holds, resulting
in a failed collateral operation or accounting mismatch — a Medium-severity
griefing / operational-DoS exploitable by any investor-initiated multicall.

## Vulnerability Details
File: periphery-v3/contracts/rwa/SecuritizeWallet.sol

_addCollateral is called from within multicall() when the call selector
matches ICreditFacadeV3Multicall.addCollateral or addCollateralWithPermit:

    function _addCollateral(
        address token,
        address investor,
        uint256 amount
    ) internal {
        ERC20(token).safeTransferFrom(investor, address(this), amount);
        ERC20(token).forceApprove(address(_CREDIT_MANAGER), amount);
    }

The flow:
  1. safeTransferFrom pulls `amount` from the investor.
  2. For a fee-on-transfer token, actual received = amount - fee < amount.
  3. forceApprove grants the CreditManager an allowance of `amount`.
  4. When the CreditManager subsequently calls transferFrom(wallet, ...,
     amount), the wallet only holds (amount - fee) — the call either:
     (a) reverts because the wallet's balance is insufficient; or
     (b) the CreditManager's collateral accounting records `amount` but only
         `amount - fee` was deposited, creating a permanent mismatch.

This path is reachable through the investor-facing multicall on the factory:

    function multicall(
        address creditAccount,
        MultiCall[] calldata calls,
        ...
    ) external override {
        if (msg.sender != creditAccountInfo.investor) revert ...;
        ...
        _multicall(creditAccountInfo.wallet, calls);
    }

No pre/post balance check is performed to reconcile actual received amount.

## Impact Details
- Collateral addition silently mints the wrong collateral credit to the
  investor's credit account (over-crediting or failing to credit).
- CreditManager may be left with an over-approved but under-funded allowance,
  which downstream logic may exploit or which causes unexpected reverts.
- Severity: Medium — operational DoS and potential collateral accounting
  discrepancy for investors using fee-on-transfer tokens.

MultiCall[] memory calls = new MultiCall[](1);
calls[0] = MultiCall({
